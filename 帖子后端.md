# 帖子后端技术方案

## 一、结论

帖子 / 论坛模块后端一期采用以下技术栈：

- Web 框架：FastAPI
- 数据库：PostgreSQL
- ORM：SQLAlchemy 2.x
- 数据库迁移：Alembic
- 鉴权：JWT Bearer Token
- 密码哈希：Argon2 或 bcrypt
- 对象存储：预留，建议 OSS / S3 兼容服务
- CI/CD：GitHub Actions
- 部署：Linux + Docker

这套组合适合当前移动端 API 场景，也能支撑后续评论、关注、搜索、审核、推荐等扩展。

---

## 二、一期目标

本期后端目标不是做完整社区平台，而是稳定支撑当前 Flutter 前端中的帖子列表、帖子详情、评论回复、点赞、关注流和发布能力。

一期必须覆盖：

- 用户注册、登录、获取当前用户
- Newest / Hot / Following 三种帖子流
- 帖子详情
- 发布帖子
- 帖子点赞 / 取消点赞
- 评论列表
- 发表评论 / 回复评论
- 评论点赞 / 取消点赞
- 计数一致性
- 基础分页

一期可选但建议预留：

- 删除帖子
- 删除评论
- 图片上传
- 举报 / 拉黑
- 话题、投票

## 三、为什么这样选型

### 1. FastAPI

选 FastAPI 的原因：

- 更适合纯 API 项目
- 类型注解和 Pydantic 对接口约束清晰
- 自动生成 OpenAPI 文档，适合前后端联调
- 对后续拆分 service、依赖注入、鉴权扩展比较自然

### 2. PostgreSQL

选 PostgreSQL 的原因：

- 关系型建模适合用户、帖子、评论、点赞、关注
- 支持复杂查询和索引优化
- 后续可接全文搜索、JSON 扩展、地理位置扩展
- 对计数、排序、事务一致性支持更稳

### 3. SQLAlchemy 2.x + Alembic

- SQLAlchemy 2.x 适合做清晰的领域模型和查询封装
- Alembic 负责 schema 版本管理，避免人工改表

### 4. GitHub Actions

GitHub Actions 只负责：

- Lint
- Test
- Build
- Deploy 流程触发

GitHub Actions 不是正式运行环境，服务本体仍然部署在 Docker 或 Linux 主机上。

## 五、核心数据模型

一期建议至少包含以下表：

- `users`
- `posts`
- `post_media`
- `post_likes`
- `comments`
- `comment_likes`
- `user_follows`

### 1. users

字段建议：

- `id` bigint / uuid，主键
- `username` varchar，唯一
- `nickname` varchar
- `avatar_url` varchar，可空
- `password_hash` varchar
- `status` smallint，默认正常
- `created_at` timestamptz
- `updated_at` timestamptz

约束建议：

- `username` 唯一索引
- `status` 用于后续封禁、注销、冻结

### 2. posts

字段建议：

- `id`
- `author_id`
- `content_text` text，可空
- `location_name` varchar，可空
- `topic_text` varchar，可空，一期先做单字段预留
- `media_type` varchar，`image` / `video` / `mixed_reserved`
- `like_count` integer，默认 0
- `comment_count` integer，默认 0
- `view_count` integer，默认 0
- `is_deleted` boolean，默认 false
- `created_at`
- `updated_at`

说明：

- 一期只开放图片帖子和纯文字帖子
- `media_type` 先只允许 `image` 或空
- `comment_count` 必须包含 1/2/3 级评论总数
- `view_count` 一期可做简单累加，不做严格去重

索引建议：

- `(created_at desc, id desc)` 支撑 Newest
- `(like_count desc, comment_count desc, created_at desc, id desc)` 支撑 Hot
- `(author_id, created_at desc)` 支撑用户主页和关注流拼接
- `where is_deleted = false` 的部分索引可在后期加

### 3. post_media

字段建议：

- `id`
- `post_id`
- `media_type`，一期固定 `image`
- `url`
- `thumbnail_url`，可空
- `sort_order`
- `width`，可空
- `height`，可空
- `created_at`

约束建议：

- 一个帖子最多 9 条图片记录，由应用层校验
- 唯一约束：`(post_id, sort_order)`

### 4. post_likes

字段建议：

- `id`
- `post_id`
- `user_id`
- `created_at`

约束建议：

- 唯一约束：`(post_id, user_id)`

用途：

- 防止重复点赞
- 取消点赞时按唯一键删除

### 5. comments

字段建议：

- `id`
- `post_id`
- `user_id`
- `parent_comment_id`，可空
- `root_comment_id`，可空
- `reply_to_user_id`，可空
- `content` text
- `level` smallint，只允许 1 / 2 / 3
- `like_count` integer，默认 0
- `is_deleted` boolean，默认 false
- `created_at`
- `updated_at`

说明：

- 一级评论：`parent_comment_id = null`，`root_comment_id = null`，`level = 1`
- 二级评论：`parent_comment_id = 一级评论ID`，`root_comment_id = 一级评论ID`，`level = 2`
- 三级评论：`parent_comment_id = 二级或三级评论ID`，`root_comment_id = 一级评论ID`，`level = 3`
- 超过三级时，不新增第四级，仍写成 `level = 3`
- 超过三级时，`parent_comment_id` 指向当前被回复评论，`root_comment_id` 保持一级评论 ID
- 前端用 `reply_to_user_id` 补出“Reply to xxx”

这样既满足“最多三级展示”，又不会丢失回复关系。

索引建议：

- `(post_id, created_at asc, id asc)`
- `(root_comment_id, created_at asc, id asc)`
- `(user_id, created_at desc)`

### 6. comment_likes

字段建议：

- `id`
- `comment_id`
- `user_id`
- `created_at`

约束建议：

- 唯一约束：`(comment_id, user_id)`

### 7. user_follows

字段建议：

- `id`
- `follower_user_id`
- `followed_user_id`
- `created_at`

约束建议：

- 唯一约束：`(follower_user_id, followed_user_id)`
- 校验：`follower_user_id != followed_user_id`

## 六、和前端字段的对齐规则

当前 Flutter 端帖子模型需要这些字段，见 `lib/models/post_model.dart`：

- 帖子：`id / userId / userName / userAvatar / createdAt / content / media / location / likes / liked / views / isHot / isFollowingAuthor / comments`
- 评论：`id / userId / userName / content / createdAt / parentId / replyToName / level / replies`

因此后端响应层必须补齐以下派生字段，而不是只返回裸表字段：

### 帖子响应 DTO

- `postId`
- `userId`
- `userName`
- `userAvatar`
- `contentText`
- `mediaList`
- `locationName`
- `publishTime`
- `likeCount`
- `commentCount`
- `viewCount`
- `isLiked`
- `isHot`
- `isFollowed`
- `isMine`

### 评论响应 DTO

- `commentId`
- `postId`
- `parentCommentId`
- `rootCommentId`
- `replyToUserId`
- `replyToUserName`
- `userId`
- `userName`
- `userAvatar`
- `content`
- `level`
- `likeCount`
- `isLiked`
- `createTime`

结论：

- `isLiked`、`isFollowed`、`isMine` 不一定落库
- 这些字段应该根据“当前登录用户 + 业务表”动态计算
- 后端不能只交付表结构，必须交付稳定 DTO

## 七、计数与一致性策略

这是一期必须提前定下来的部分。

### 1. 点赞数

策略：

- `post_likes` 存点赞明细
- `posts.like_count` 存冗余计数
- 点赞成功后事务内 `+1`
- 取消点赞成功后事务内 `-1`

原因：

- 列表页和详情页都需要高频读取点赞数
- 每次聚合 `count(*)` 成本不必要

### 2. 评论数

策略：

- `comments` 存明细
- `posts.comment_count` 存冗余计数
- 新增评论或回复后事务内 `+1`
- 删除有效评论后事务内 `-1`
- 这里的总数必须包含 1/2/3 级评论

### 3. 浏览量

一期建议：

- `posts.view_count` 直接累加
- 每次进入详情页时调用一次浏览接口或详情内完成一次累加
- 暂不做用户维度去重

### 4. 一致性要求

- 所有点赞、取消点赞、发表评论、删除评论操作都在事务内完成
- 明细表写成功但计数字段未更新，必须回滚
- 列表和详情读取同一张 `posts` 表中的计数字段

这样才能满足前端文档中的“列表页与详情页一致”要求。

## 八、分页与排序方案

一期建议统一使用游标分页，不建议 offset 分页。

### 1. 帖子列表

接口建议：

`GET /api/v1/posts?feed=newest|hot|following&cursor=xxx&limit=20`

参数约定：

- `feed`
  - `newest`
  - `hot`
  - `following`
- `cursor`
  - 首次请求不传
  - 后续传服务端返回的下一页游标
- `limit`
  - 默认 20
  - 最大 50

排序规则：

- `newest`：`created_at desc, id desc`
- `hot`：`like_count desc, comment_count desc, created_at desc, id desc`
- `following`：只查我关注作者的帖子，排序同 `newest`

### 2. 评论列表

接口建议：

`GET /api/v1/posts/{post_id}/comments?cursor=xxx&limit=30`

返回建议：

- 一级评论按时间正序
- 每条一级评论下附带其二级、三级回复
- 或者返回平铺结构 + `rootCommentId`，由前端组装

一期为了贴合当前前端模型，建议直接返回树形结构。

### 2. 帖子接口

- `GET /api/v1/posts`
- `GET /api/v1/posts/{post_id}`

### 4. 关注流接口

帖子流已经可以通过 `feed=following` 支撑。

- `POST /api/v1/users/{user_id}/follow`
- `DELETE /api/v1/users/{user_id}/follow`

### 5. 上传接口

一期如果要做真实图片上传，建议增加：

- `POST /api/v1/uploads/images`

如果当前只是联调，可先使用 mock URL，不阻塞帖子主链路。

## 十、评论三级规则的最终定义

这是最容易返工的部分，这里直接定死。

### 1. 一级评论

- 直接评论帖子
- `level = 1`
- `parent_comment_id = null`
- `root_comment_id = null`

### 2. 二级评论

- 回复一级评论
- `level = 2`
- `parent_comment_id = 一级评论ID`
- `root_comment_id = 一级评论ID`

### 3. 三级评论

- 回复二级评论
- `level = 3`
- `root_comment_id = 一级评论ID`

### 4. 超过三级

- 不创建 `level = 4`
- 仍保存为 `level = 3`
- `parent_comment_id` 指向当前被回复评论
- `root_comment_id` 仍指向一级评论
- `reply_to_user_id` 存被回复人

这样做的结果：

- 数据库最大层级固定为 3
- 前端仍能显示“回复某某”
- 业务上不会丢掉真实回复目标

## 十一、鉴权与安全边界

一期鉴权方案：

- 登录成功签发 `access token`
- 前端通过 `Authorization: Bearer <token>` 访问受保护接口
- 密码仅存哈希值

一期建议补充但可以第二阶段实现：

- refresh token
- 登录失败限流
- IP / 用户维度防爆破
- 注销与 token 失效
- 图片上传鉴权

结论：

只做 `access token` 可以支持一期联调，但文档中必须明确这是阶段性方案，不是最终安全方案。

## 十二、错误码建议

至少统一以下错误码语义：

- `400` 参数错误
- `401` 未登录或 token 无效
- `403` 无权限，例如删除他人帖子
- `404` 帖子或评论不存在
- `409` 重复操作，例如重复点赞、重复关注、用户名已存在
- `422` 请求格式合法但业务校验失败
- `500` 服务内部错误

建议统一响应格式：

```json
{
  "code": "POST_NOT_FOUND",
  "message": "Post does not exist",
  "requestId": "trace-id"
}
```

## 十四、一期验收标准

后端一期完成的标志：

- 能注册、登录并访问受保护接口
- `GET /posts` 能正确返回 `newest / hot / following`
- `GET /posts/{id}` 与列表页字段口径一致
- `POST /posts` 能发布纯文字和图文帖子
- 图片数量超过 9 张时校验失败
- 点赞后列表和详情计数一致
- 评论后列表和详情评论数一致
- 评论最多三级，超过三级不新增第四级
- 评论接口能返回前端可直接渲染的数据结构

## 十五、最终结论

最终采用的后端架构不变：

- FastAPI
- PostgreSQL
- SQLAlchemy 2.x
- Alembic
- JWT Bearer Token
- GitHub Actions
- Linux / Docker

但真正可开工的关键不只是技术栈，而是以下五点已经明确：

- 表结构
- DTO 字段
- 分页排序
- 评论三级规则
- 计数一致性策略

按这份方案推进，一期后端可以直接对齐当前 Flutter 前端需求并进入实现阶段。
